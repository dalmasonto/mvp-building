# PRD: MVP Building Beginner Masterclass

## 1. Overview

**Product Name:** MVP Building Beginner Masterclass
**Instructor:** Dalmas
**Format:** Web-based course (hosted on a dedicated website)
**Audience:** Beginner founders, developers, and builders who want to move from ideas to shipping real products.

**Core Objective:**
Teach beginners how to go from **idea → MVP → first users** with the correct mentality, work ethic, and practical execution skills.

This masterclass prioritizes **execution over theory**. The curriculum focuses on how real builders structure ideas, validate them quickly, build minimal versions, and learn from users.

**Key Philosophy**

1. Ideas are cheap; execution is the differentiator.
2. Speed matters more than perfection.
3. The first version should be ugly but functional.
4. Learn from users immediately.
5. Build systems that allow consistent output.

### Topics

**2.1 Sources of Good Ideas**

Common sources:

* Personal pain points
* Developer tools
* Workflow inefficiencies
* Market gaps
* Communities

**2.2 Idea Evaluation Framework**

A simple scoring model:

| Factor           | Description                |
| ---------------- | -------------------------- |
| Problem severity | How painful is the problem |
| Market access    | Can you reach users easily |
| Build complexity | Can you build v1 quickly   |
| Monetization     | Is someone willing to pay  |

Validation methods:

* Talking to users
* Twitter/X posts

* Community feedback

## Section 3 — MVP Thinking

### Objective

Help students understand what an MVP actually is.

### Topics

**3.1 What is an MVP**

Definition:

> The smallest functional version of a product that solves the core problem.

**3.2 What an MVP is NOT**

* Fully polished product
* Feature complete platform
* Scalable architecture
* Perfect UI

**3.3 MVP Design Framework**

Step process:

1. Define problem
2. Define single user
3. Define core action
4. Remove everything else

Example:

Speed camera app

Core MVP:

* Map
* Camera locations
* Distance alert

Not needed initially:

* Accounts
* Social features
* Analytics
* Gamification

**3.4 Feature Cutting Framework**

Questions:

* Does the feature solve the main problem?
* Can users get value without it?
* Does it delay shipping?

If yes → cut it.

## Section 4 — Work Ethic

### Objective

Teach consistent building habits.

### Topics

**4.1 Builder Work Cycles**

Recommended cadence:

Idea → Build → Ship → Feedback → Iterate

Cycle length:

* MVP build: 1–4 weeks
* Iteration cycles: 3–7 days

**4.2 Daily Builder Schedule**

Example:

Morning

* Deep building work

Afternoon

* Debugging
* UI improvements

Evening

* User feedback
* Documentation

**4.3 Avoiding Builder Burnout**

Key principles:

* Ship small
* Celebrate small wins
* Take breaks

**4.4 Consistency Over Intensity**

Better:

* 2 hours daily for 6 months

Than:

* 14 hours daily for 2 weeks then quitting.
